## Review Metadata

## Executive Summary

The repository is in a workable state: the current test suite passes, the CLI is coherent, and the skills share a consistent output contract. The main issue is not basic stability but output quality and trustworthiness. Several skills produce results that look plausible but are driven by shallow string matching or hardcoded templates. That makes the toolkit vulnerable to false positives, misleading recommendations, and low-value outputs on real repositories.

The highest-priority improvements are:

1. Fix false positives in `code_reviewer`.
2. Fix ranking and false risk notes in `test_generator`.
3. Fix Unicode output-name collisions in `core/io.py`.
4. Reduce misleading hardcoded output in `architecture_designer` and `figma_ui_architect`.
5. Expand tests to verify output quality, not only file creation and section presence.

## Findings

### 1. `code_reviewer` produces false positives from plain text and string literals

Severity: High

Files:

- `src/ai_skills_toolkit/skills/code_reviewer/skill.py`

Relevant areas:

- `_extract_findings(...)`
- checks for `eval(`, `TODO`, `FIXME`, `assert `

Problem:

The reviewer scans raw lines and flags patterns with naive substring matching. It does not distinguish between executable code and text inside strings, comments, markdown snippets, or examples. As a result, the skill reports issues even when dangerous tokens appear only as text.

Observed behavior:

- The skill reports `Use of eval` on its own implementation because the rule text itself contains `eval(`.
- It reports `Unresolved TODO/FIXME` based on explanatory strings and report templates.
- It reports `Runtime assert in non-test module` when the word `assert` appears in example text.

Impact:

- Trust in the tool degrades quickly.
- Users may spend time investigating non-issues.
- The skill can generate noisy reports against its own codebase and likely against documentation-heavy repos.

What to fix:

- Replace line-level substring checks with Python-aware parsing using `tokenize` or `ast`.
- Ignore strings and comments when searching for executable patterns such as `eval`, `print`, `assert`, and bare `except`.
- Keep heuristic mode if desired, but label it clearly and add deduplication or confidence scoring.
- Add regression tests where suspicious tokens appear only in strings/comments and must not be reported.

Suggested implementation direction:

- Use `ast.walk()` to detect:
  - `ast.Call` to `eval`
  - `ast.Assert`
  - `ast.ExceptHandler` with `type is None`
- Use `tokenize` for TODO/FIXME only in comments, not in arbitrary strings.

### 2. `test_generator` ranks low-value files ahead of meaningful targets

Severity: High

Files:

- `src/ai_skills_toolkit/skills/test_generator/skill.py`

Relevant area:

- target sorting in `generate_test_plan(...)`

Problem:

The ranking expression combines `-p.stat().st_size` with `reverse=True`. This inverts the intended behavior and tends to push smaller files upward. On the current repository, schema files are prioritized ahead of more behavior-heavy modules.

Impact:

- The test plan highlights poor targets first.
- Generated plans are less actionable.
- The user may believe the tool found the most important coverage opportunities when it did not.

What to fix:

- Remove the double inversion and make ranking explicit.
- Prefer larger or more behavior-rich modules after focus-path priority is applied.
- Consider a better scoring function that uses:
  - public function count
  - class count
  - branching/complexity hints
  - I/O usage
  - presence of error handling

Suggested implementation direction:

- Sort by descending priority, then descending “test value” score.
- A simple first step:
  - `key=lambda p: (_priority(...), p.stat().st_size)`
  - `reverse=True`
- A better next step:
  - compute a score from AST-derived signals instead of raw file size only.

### 3. `test_generator` emits false `risk_notes` from arbitrary text

Severity: High

Files:

- `src/ai_skills_toolkit/skills/test_generator/skill.py`

Relevant area:

- `_risk_notes(...)`

Problem:

Risk notes are currently generated by checking whether substrings such as `except:`, `TODO`, `FIXME`, and `pass` appear anywhere in file content. This is too naive. Even a string like `bypass auth` triggers the `pass` warning.

Impact:

- The risk section becomes noisy and misleading.
- The user may overestimate the incompleteness or fragility of modules.

What to fix:

- Parse Python syntax rather than scanning raw content.
- Detect real `pass` statements via AST.
- Restrict TODO/FIXME detection to comments or doc comments.
- Add regression tests for:
  - `pass` inside unrelated words
  - TODO in string literals
  - `except:` inside sample text

### 4. Output file names collide for non-ASCII names

Severity: High

Files:

- `src/ai_skills_toolkit/core/io.py`

Relevant area:

- `slugify(...)`

Problem:

`slugify` keeps only ASCII `[a-z0-9]`. Any Cyrillic or other non-ASCII title collapses to an empty slug and becomes `output`. That means many distinct user-provided names map to the same file path.

Examples observed:

- `отчет` -> `output`
- `тест` -> `output`
- `арх-спек` -> `output`

Impact:

- Unexpected collisions between unrelated runs.
- Spurious `FileExistsError`.
- Poor ergonomics for non-English users.

What to fix:

- Support Unicode-safe slugs or transliteration.
- If transliteration is unavailable, append a stable short hash of the original input.
- Add tests for Cyrillic and mixed-language filenames.

Suggested implementation direction:

- Option A: transliterate to ASCII and preserve semantics.
- Option B: sanitize less aggressively and permit safe Unicode on modern filesystems.
- Option C: fallback to `output-<hash>` when ASCII slug becomes empty.

### 5. `architecture_designer` output is too hardcoded and can mislead users

Severity: Medium

Files:

- `src/ai_skills_toolkit/skills/architecture_designer/skill.py`

Problem:

The skill accepts product-specific inputs, but core outputs are mostly fixed templates:

- same components
- same API endpoints
- same data entities
- same risks
- same open questions

This creates the appearance of tailored architecture work while actually emitting generic boilerplate. A payroll tool and a chat app currently receive essentially the same API surface and structure.

Impact:

- The artifact looks more specific than it is.
- Users may copy assumptions that do not match their domain.
- This reduces the value of the skill as a planning tool.

What to fix:

- Make outputs depend on user requirements.
- Infer probable domain entities and API nouns from product goal and requirements.
- Emit explicit disclaimers when sections are boilerplate.
- Prefer “candidate options” over fake precision when context is insufficient.

Suggested implementation direction:

- Extract keywords from `product_goal`, `functional_requirements`, and `constraints`.
- Derive domain-specific entities and endpoint candidates from those inputs.
- Separate invariant architecture guidance from generated domain-specific sections.

### 6. `figma_ui_architect` is also largely static and only lightly parameterized

Severity: Medium

Files:

- `src/ai_skills_toolkit/skills/figma_ui_architect/skill.py`

Problem:

The skill varies the title, product goal, and a few handoff lines, but the main body is largely fixed:

- same flows
- same screens
- same components
- same states
- same open questions

For different products, the output changes less than the framing suggests.

Impact:

- The spec may feel polished while being structurally generic.
- It is less useful for real UI planning and handoff.

What to fix:

- Derive user flows and screens from the supplied JTBD and user segments.
- Make defaults clearly fall back only when inputs are missing.
- Add stronger links between constraints and recommended UI structure.

Suggested implementation direction:

- If JTBD entries are present, generate flows from them rather than using fixed sequences.
- Map constraints such as `Desktop-first`, `Tablet`, `WCAG`, `High-density data`, `Mobile-first` into screen/state/handoff guidance.

### 7. `doc_writer` reveals hidden top-level entries even though analysis hides them by default

Severity: Medium

Files:

- `src/ai_skills_toolkit/skills/doc_writer/skill.py`

Problem:

`doc_writer` calls `analyze_repository` with default hidden-file behavior, but `_collect_top_level_entries(...)` uses `repo_path.iterdir()` directly and does not filter hidden entries. That can expose `.git`, `.env`, and other hidden items even when the analysis intentionally excludes them.

Impact:

- Inconsistent behavior across skills.
- Potential leakage of sensitive filenames or internal scaffolding in generated docs.

What to fix:

- Reuse the same hidden-file policy as `repo_analyzer`.
- Optionally expose `include_hidden` as an explicit `doc_writer` input.
- Add a regression test showing hidden entries are omitted by default.

### 8. CLI and tests validate mechanics more than output quality

Severity: Medium

Files:

- `tests/test_cli.py`
- `tests/test_code_reviewer.py`
- `tests/test_test_generator.py`
- other per-skill tests

Problem:

Most tests currently check that:

- the command exits with `0`
- a file is written
- a heading exists

That is useful for smoke coverage, but it does not validate whether the report is correct, relevant, ranked properly, or free from obvious false positives.

Impact:

- Significant behavior defects can survive while all tests remain green.
- The suite gives a false sense of quality.

What to fix:

- Add negative tests for false positives.
- Add ranking tests for `test_generator`.
- Add Unicode-path and Unicode-output-name tests.
- Add hidden-file consistency tests.
- Add behavior-quality assertions, not only “contains section heading”.

## Secondary Improvements

These are not as urgent as the items above but would improve quality and maintainability.

### 9. Share repository traversal policies across skills

Current skills each define their own excluded directories or traversal behavior. That creates drift:

- `repo_analyzer` excludes `.pytest_cache`
- `code_reviewer` does not
- `test_generator` excludes `tests`
- `doc_writer` uses a separate path walk for top-level entries

Recommendation:

- Centralize repository traversal policies in one helper module.
- Allow each skill to opt into additional exclusions, but keep the defaults aligned.

### 10. Add structured metadata quality to outputs

`SkillRunResult.metadata` is currently thin and inconsistent across skills. It is enough for light reporting, but not enough for robust chaining.

Recommendation:

- Add stable metadata fields for:
  - input path
  - scan scope
  - warnings
  - truncation flags
  - heuristic mode flags
  - counts by severity/type

This will make these skills easier to orchestrate in larger agent workflows.

### 11. Distinguish “template output” from “analysis output”

Some skills are mainly analysis-based (`repo_analyzer`), while others are template-driven (`architecture_designer`, `figma_ui_architect`). Right now they are presented with similar confidence.

Recommendation:

- Explicitly tag output sections as:
  - observed from repository
  - inferred from input
  - default template

That would improve honesty and usability.

## Concrete Fix Plan

1. Refactor `code_reviewer` to use AST/token-level detection.
2. Refactor `test_generator` ranking and `risk_notes`.
3. Fix Unicode-safe output naming in `core/io.py`.

### Priority 2: reduce misleading output

4. Make `architecture_designer` outputs depend materially on user inputs.
5. Make `figma_ui_architect` derive flows/screens from JTBD and constraints.
6. Align `doc_writer` hidden-file behavior with `repo_analyzer`.

### Priority 3: test suite quality

7. Add negative tests for false positives in `code_reviewer`.
8. Add regression tests for `test_generator` ranking.
9. Add Unicode filename tests for `slugify` / output generation.
10. Add hidden-file leakage tests for `doc_writer`.

## Suggested Test Additions

### `code_reviewer`

- A file where `eval(` appears only inside a string literal: no finding expected.
- A file where `TODO` appears only in a string literal: no finding expected.
- A file with real `except:` and real `eval(...)`: findings expected.

### `test_generator`

- Ensure a larger behavior-heavy module ranks above a tiny schema-only module when focus priority is equal.
- Ensure `bypass` does not trigger `pass` risk notes.
- Ensure TODO in string literals does not create TODO risk notes.

### `core/io.py`

- Distinct Cyrillic names should produce distinct safe output paths.
- Mixed Cyrillic/Latin names should not collapse to `output`.

### `doc_writer`

- Hidden files like `.env` should be excluded by default from top-level structure.
- Optional flag can enable hidden entries when explicitly requested.

## Positive Notes

The repo also has clear strengths:

- clean package layout
- consistent `run(...)` contract across skills
- coherent CLI surface
- output overwrite protection
- readable code and straightforward module boundaries

That is a good base. The main next step is to raise the semantic quality of outputs so that the toolkit is not just operational, but dependable.

## Final Recommendation

The repository should be treated as a solid phase-1/phase-2 scaffold rather than a production-grade analysis toolkit. The infrastructure is good enough to build on, but several skills currently overstate how much real analysis they perform. The most valuable investment now is not adding more skills, but making the current ones trustworthy on realistic repositories and non-English user workflows.
